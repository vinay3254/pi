# EtherXMeet Project Explanation

This project has two main parts:

1. A React frontend inside `frontend/`
2. A Hardhat smart-contract project in the repo root

## What `frontend/dist/index.html` is

`frontend/dist/index.html` is the built output generated by Vite after running the frontend build.

It is not the file you usually edit by hand. Its job is to:

- define the basic HTML document
- load fonts and metadata
- provide `<div id="root"></div>` as the mount point
- load the compiled JS and CSS bundles from `/assets/...`

That means the real app UI is not written inside this HTML file. React injects the app into the `root` div at runtime.

## What the source HTML is

The source version is `frontend/index.html`.

That file is used during development and looks almost the same, but instead of loading a compiled asset file, it loads:

`/src/main.jsx`

So the flow is:

`frontend/index.html` -> `frontend/src/main.jsx` -> `App.jsx`

After build, Vite transforms that into:

`frontend/dist/index.html` -> `/assets/index-*.js`

## What `frontend/src/main.jsx` does

`main.jsx` is the React entry point.

It:

- finds the DOM node with `id="root"`
- creates a React root with `ReactDOM.createRoot(...)`
- wraps the app in `React.StrictMode`
- wraps the app in `HMSRoomProvider` from `@100mslive/react-sdk`
- renders `<App />`

So this is the file that actually starts the frontend application.

## What the smart contract does

`contracts/MeetingRegistry.sol` stores meeting metadata on-chain.

Each stored meeting includes:

- `meetingId`
- `host`
- `startTime`
- `endTime`
- `participantCount`
- `notesHash`

Important rules in the contract:

- `meetingId` cannot be empty
- a meeting ID can only be stored once
- `startTime` must be non-zero
- `endTime` must be greater than or equal to `startTime`

It also keeps a per-host history using `hostMeetings`.

## What the test file does

`test/MeetingRegistry.test.js` checks that the contract behaves correctly.

It verifies:

- deployment works
- `storeMeeting` emits the right event
- saved meeting data can be read back
- invalid inputs revert with the expected messages
- duplicate meeting IDs are rejected
- meetings are tracked separately per host
- unknown meeting IDs return a zero-value struct

## What the deploy script does

`scripts/deploy.js` deploys `MeetingRegistry` using Hardhat and then writes the deployed contract address and ABI into:

`frontend/src/contracts/MeetingRegistry.json`

That lets the frontend talk to the deployed contract without hardcoding ABI details manually.
