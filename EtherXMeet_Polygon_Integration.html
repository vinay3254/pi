1. Project Overview

EtherXMeet is a decentralised Web3 video conferencing platform built on the Polygon blockchain. Every meeting event — creation, participant join, chat message, and meeting end — is recorded permanently on-chain as a tamper-proof audit trail. At the end of each meeting, the host receives an IPFS receipt and an ERC-721 NFT as a verifiable proof of the session.

Core Value Proposition

  • On-chain meeting records — every session is stored in MeetingRegistry.sol on Polygon
  • IPFS receipts — meeting metadata pinned to IPFS via Pinata at end of session
  • NFT receipts (ETXR) — ERC-721 token minted to the host wallet as proof of meeting
  • LiveKit video rooms — real-time video via LiveKit cloud (wss://etherxmeet-9kvqnuhc.livekit.cloud)
  • Screen recording — host can record the session, uploaded to backend storage

3. Smart Contracts

3.1 MeetingRegistry.sol

Deployed at: 0x5FbDB2315678afecb367f032d93F642f64180aa3 (Hardhat local)
File: contracts/MeetingRegistry.sol

This is the core contract. It stores every meeting's full lifecycle on-chain and is the single source of truth for all session data.

Data Structure

struct Meeting {
    string   meetingId;        // Unique room code e.g. "etherx-ua8ndzdk"
    address  host;             // Wallet address of the meeting host
    uint256  startTime;        // Unix timestamp — meeting started
    uint256  endTime;          // Unix timestamp — meeting ended (0 = still live)
    uint256  participantCount; // Number of wallets that joined
    bytes32  notesHash;        // keccak256 hash of AI notes
    string   metadataCID;      // IPFS CID for rich metadata JSON
}

Key Functions

FunctionWho Calls ItWhat It Does
createMeeting(meetingId)Host walletRegisters a new meeting on-chain, sets startTime, stores host address
joinMeeting(meetingId)Participant walletAdds caller to participants list, increments participantCount
sendMessage(meetingId, contentHash)Any participantRecords a keccak256 hash of a chat message on-chain
endMeeting(meetingId)Host wallet onlySets endTime, closes the meeting — reverts if called by non-host
getMeeting(meetingId)Anyone (read-only)Returns full Meeting struct
getMeetingsByHost(address)Anyone (read-only)Returns array of meeting IDs hosted by a given wallet
storeMeeting(...)LegacyBackward-compatible single atomic write (used before the multi-step flow)

3.2 MeetingNFT.sol (ERC-721)

Deployed at: 0xe7f1725E7734CE288F8367e1Bb143E90bb3F0512 (Hardhat local)
File: contracts/MeetingNFT.sol
Token name: EtherXMeet Receipt   Symbol: ETXR

Mints one ERC-721 NFT per ended meeting. The token URI points to the IPFS receipt JSON (ipfs://<cid>). Only the original meeting host can mint, verified on-chain by calling back into MeetingRegistry.

Key Functions

FunctionDescription
mint(meetingId, ipfsCid)Mints one ETXR token. Verifies: meeting exists, meeting has ended, caller is host, not already minted. Sets tokenURI = ipfs://<cid>.
tokenURI(tokenId)Returns the IPFS URI for a given token (ERC-721 standard)
minted(meetingId)Returns true if an NFT has already been minted for that meeting
tokenOfMeeting(meetingId)Returns the tokenId for a given meetingId

Mint Logic

function mint(string calldata meetingId, string calldata ipfsCid)
    external returns (uint256 tokenId)
{
    require(!minted[meetingId], "already minted");
    IMeetingRegistry.Meeting memory m = IMeetingRegistry(registry).getMeeting(meetingId);
    require(m.startTime != 0,   "meeting does not exist");
    require(m.endTime   != 0,   "meeting not ended yet");
    require(m.host == msg.sender, "only host can mint receipt");

    tokenId = _nextTokenId++;
    minted[meetingId] = true;
    tokenOfMeeting[meetingId] = tokenId;
    _safeMint(msg.sender, tokenId);
    _setTokenURI(tokenId, string(abi.encodePacked("ipfs://", ipfsCid)));
    emit ReceiptMinted(meetingId, msg.sender, tokenId, ipfsCid);
}

const { account, balance, signer, isConnecting, connectError, connect } = useWallet();

On connect, the context automatically requests a network switch to the correct chain based on VITE_NETWORK:

// VITE_NETWORK=localhost → chainId 0x7a69 (31337, Hardhat)
// VITE_NETWORK=amoy     → chainId 0x13882 (80002, Polygon Amoy)
// VITE_NETWORK=mainnet  → chainId 0x89 (137, Polygon)

Native currency is set correctly per network — MATIC for Polygon/Amoy, ETH for Hardhat.

5.2 useMeetingContract Hook

File: frontend/src/hooks/useMeetingContract.js

Wraps all MeetingRegistry contract calls with loading/error state:

const {
  createMeeting, joinMeeting, sendMessage, endMeeting,
  getMeeting, getMeetingsByHost,
  isTxPending, txHash, error
} = useMeetingContract();

Uses ethers.Contract with the signer from WalletContext. The contract ABI and address are imported from frontend/src/contracts/MeetingRegistry.json (written by the deploy script).

Transaction Flow

  1. User calls e.g. createMeeting(roomCode)
  2. MetaMask popup appears for the user to sign + approve gas
  3. isTxPending = true — ChainHUD shows a spinner
  4. Hook waits for 1 confirmation (tx.wait(1))
  5. Returns the transaction receipt; isTxPending = false

5.3 useMeetingNFT Hook

File: frontend/src/hooks/useMeetingNFT.js

const { mintReceipt, isMinting, mintError } = useMeetingNFT();

// Call after endMeeting + IPFS pin:
const { tokenId, txHash } = await mintReceipt(meetingId, ipfsCid);

Wraps the MeetingNFT.mint() call. Returns the minted tokenId parsed from the ReceiptMinted event. Non-blocking — if minting fails (e.g. wallet disconnected), the receipt modal still opens without the NFT badge.

5.4 ChainHUD Overlay

File: frontend/src/components/web3/ChainHUD.jsx

A floating overlay shown inside the meeting room that displays:

  • Connected wallet address (truncated)
  • Current network label (HARDHAT / AMOY TESTNET / POLYGON)
  • Live block number (polls every 12 seconds)
  • A spinning indicator when a transaction is pending (txPending prop)

5.5 Meeting Receipt Modal

File: frontend/src/components/web3/MeetingReceiptModal.jsx

Shown to the host after endMeeting completes. Displays:

  • Transaction hash (links to block explorer)
  • Block number
  • Meeting duration in minutes
  • Participant count (from on-chain data)
  • IPFS CID (links to Pinata gateway)
  • NFT token ID badge ("ETXR #N") — shown only if minting succeeded

5.6 OnChainConfirmModal

File: frontend/src/components/web3/OnChainConfirmModal.jsx

Shown before navigating to the room. Explains to the user that creating/joining the meeting will trigger a blockchain transaction, then calls createMeeting() or joinMeeting() on approval.

5.7 WalletBanner

File: frontend/src/components/web3/WalletBanner.jsx

Shown at the top of the Dashboard and Landing page. Shows connected wallet address + balance when connected, or a "Connect MetaMask" button when not connected.

6. IPFS Receipt Storage (Pinata)

File: frontend/src/utils/ipfs.js

After endMeeting returns a transaction receipt, the frontend pins a JSON receipt to IPFS via Pinata's REST API:

// Receipt JSON structure:
{
  "meetingId": "etherx-ua8ndzdk",
  "txHash": "0xabc123...",
  "blockNumber": 42,
  "durationMin": 23,
  "endedAt": 1714000000000
}

The returned IPFS CID (Qm... or bafy...) is:

  1. Displayed in the Meeting Receipt Modal with a gateway link
  2. Stored in metadataCID on-chain (optional — for lookup later)
  3. Passed to MeetingNFT.mint() as the token URI

VITE_PINATA_GATEWAY=https://gateway.pinata.cloud/ipfs

The frontend uses this token with @livekit/components-react's LiveKitRoom to connect to the LiveKit cloud server.

Recording Flow

  1. Host clicks Record in the meeting room
  2. Browser asks permission to share a screen/tab (getDisplayMedia)
  3. MediaRecorder captures the stream in 1-second chunks
  4. Host clicks Stop Rec (or stops screen share)
  5. A .webm blob is assembled and POSTed to /api/recordings/upload with JWT auth
  6. Backend stores file to backend/uploads/ and creates a MongoDB record
  7. Recording appears in /recordings page, playable via the stream endpoint

8. End-to-End Meeting Flows

8.1 Host Creates and Ends a Meeting

1. Landing page → "New meeting" button
   → generateRoomCode() → "etherx-xxxxxxxx"

2. OnChainConfirmModal opens
   → User approves → createMeeting(roomCode) called
   → MetaMask popup → user signs tx
   → MeetingRegistry.createMeeting() emits MeetingCreated event
   → sessionStorage: etherx_host_room = roomCode

3. Navigate to /room/etherx-xxxxxxxx
   → Jitsi iframe loads (full-screen video)
   → ChainHUD overlay shows wallet + block number
   → "Record" + "End Meeting" buttons visible (host only)

4. Host clicks "End Meeting"
   → endMeeting(roomCode) → MetaMask sign
   → MeetingRegistry.endMeeting() emits MeetingEnded

5. Parallel: pinJSON(receipt) → Pinata IPFS → returns CID

6. mintReceipt(roomCode, cid) → MeetingNFT.mint()
   → MetaMask sign → ETXR #N minted to host wallet

7. MeetingReceiptModal opens:
   ┌─────────────────────────────────┐
   │ tx: 0xabc...                    │
   │ Block: 42                       │
   │ Duration: 23 min                │
   │ Participants: 3                 │
   │ IPFS: Qm... [view]              │
   │ NFT Receipt: ETXR #7            │
   └─────────────────────────────────┘

8.2 Participant Joins a Meeting

1. Participant receives room URL: /room/etherx-xxxxxxxx
   (without ?role=host in sessionStorage)

2. If wallet connected → OnChainConfirmModal
   → joinMeeting(roomCode) → MetaMask sign
   → MeetingRegistry.joinMeeting() — participantCount++

3. Navigate to /room/etherx-xxxxxxxx
   → Jitsi iframe, no "End Meeting" button visible

8.3 Chat Messages On-Chain

1. User types in the room chat
2. sendMessage(roomCode, keccak256(messageText)) called
3. MetaMask signs the tx
4. MessageSent event emitted on-chain
5. Message content is NOT stored on-chain — only its hash
   (content stays in the Jitsi session, hash is the on-chain proof)

Known Issues

  • If the Hardhat node is restarted, contracts must be re-deployed and MetaMask's nonce may need to be reset (Settings → Advanced → Clear activity tab data)
  • Recording upload requires the user to be logged in via the backend — anonymous users' recordings are silently discarded
  • The Jitsi (8x8.vc) iframe is a third-party embed. The video quality and features depend on Jitsi's free tier
